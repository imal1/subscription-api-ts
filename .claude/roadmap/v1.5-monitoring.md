# v1.5 — 监控 & 告警

**状态：待开发**

## 版本目标

为服务添加可观测性：运行指标采集、日志聚合、异常告警，让运维不再靠"手动 SSH + journalctl"。

## 核心功能

### 1. 运行指标采集

- **指标端点**：新增 `GET /api/metrics`，返回 Prometheus 格式的指标
- **核心指标**：

| 指标 | 类型 | 说明 |
|------|------|------|
| `subscription_nodes_total` | Gauge | 当前可用节点总数 |
| `subscription_updates_total` | Counter | 订阅更新总次数 |
| `subscription_update_duration_seconds` | Histogram | 单次更新的耗时分布 |
| `subscription_source_errors_total` | Counter | 订阅源获取失败次数（按源标签） |
| `http_requests_total` | Counter | HTTP 请求总数（按端点、状态码） |
| `http_request_duration_seconds` | Histogram | HTTP 请求耗时分布 |
| `mihomo_available` | Gauge | mihomo 是否可用（0/1） |
| `singbox_available` | Gauge | sing-box 是否可用（0/1） |

### 2. 仪表盘监控面板

在仪表盘新增"监控"区域：

- **实时指标卡片**：今日更新次数、今日失败次数、平均更新耗时
- **节点趋势图**：最近 7 天节点数量变化（可选：使用简单的 SVG chart 或轻量库）
- **更新历史表**：最近 20 次更新的时间、结果、节点数、耗时
- **错误日志摘要**：最近 10 条错误日志，点击展开详情

### 3. 日志增强

- **结构化日志**：确保所有日志包含 `timestamp`、`level`、`service`、`traceId`
- **请求日志**：每个 HTTP 请求记录方法、路径、状态码、耗时、IP
- **日志轮转**：Winston 已配置按大小轮转（5MB × 5 文件），保持不变
- **日志搜索**：在仪表盘提供简单的日志搜索（按级别、时间范围、关键词）

### 4. 告警通知

- **告警规则**：

| 规则 | 条件 | 级别 |
|------|------|------|
| 订阅源全部失败 | 所有源连续 3 次获取失败 | 严重 |
| 节点数为 0 | 更新后可用节点 = 0 | 严重 |
| 单源失败 | 某个源连续 3 次获取失败 | 警告 |
| 服务重启 | systemd 服务重启 | 信息 |
| 磁盘空间不足 | `backup/` 目录 > 1GB | 警告 |

- **通知渠道**（按优先级实现）：
  1. 仪表盘内通知（内置）
  2. Webhook（可配置 URL，支持自定义模板）
  3. 邮件（后续版本）

### 5. 健康检查增强

`/api/health` 增加深度检查：

```json
{
  "status": "healthy",
  "timestamp": "...",
  "uptime": 12345,
  "version": "1.5.0",
  "checks": {
    "mihomo": { "status": "ok", "version": "v1.19.27" },
    "singbox": { "status": "ok" },
    "disk": { "status": "ok", "free": "8.2GB" },
    "subscription": { "status": "ok", "nodes": 5, "lastUpdate": "..." }
  }
}
```

## 验收标准

- [ ] `/api/metrics` 返回 Prometheus 格式指标
- [ ] 仪表盘监控面板显示实时指标和历史数据
- [ ] 更新历史表显示最近 20 次更新记录
- [ ] 告警规则触发后在仪表盘显示通知
- [ ] `/api/health` 深度检查包含所有子服务状态
- [ ] 请求日志记录完整（方法、路径、状态码、耗时）
- [ ] 日志搜索功能可按级别和时间过滤
